\section{Threats to Validity}
\label{sec:threats}

\textbf{Language specificity.}
Both implementations are in Go.  The relative performance of
multiprecision vs.\ fixed-width digit generation may differ in languages
with different allocation models (e.g., C/Rust with stack allocation, or
JVM with generational GC).  The \texttt{sync.Pool} optimization in the
\burgerdybvig implementation is Go-specific.

\textbf{Single-architecture performance data.}
Performance benchmarks run on a single x86-64 machine.
Cross-compilation \emph{determinism} is verified by building arm64
binaries and executing them under QEMU user-mode emulation on the x86-64
host.  Because QEMU user-mode translates system calls but executes
floating-point operations on the host CPU, this verifies Go compiler
determinism across target architectures, not native arm64 hardware
behavior.  Arm64 \emph{performance} benchmarks are deferred due to lack
of bare-metal hardware.  Future work should include bare-metal arm64
benchmarks on Graviton, Apple Silicon, or Ampere hardware.

\textbf{Single JavaScript engine oracle.}
Oracle vectors are generated from V8 (via Node.js).  While ECMA-262
\texttt{Number::toString} semantics are normative and engine-independent,
our vectors are verified against only one engine and one version
(V8~12.9.202.28).  Cross-version stability is expected from the
normative spec but not independently verified.  This is mitigated by
dense boundary coverage: 286{,}362 vectors including all subnormals
up to $2^{12}$, all binary64 powers of two, subnormal/normal boundary
neighborhoods, \rfcjcs Appendix~B examples with neighbors, and
pseudorandom fill.

\textbf{Workload representativeness.}
Benchmark workloads are synthetic, designed to bracket the performance
space between number-heavy and string-dominant extremes.  Real-world JSON
payloads may have different number density, nesting depth, and key
length distributions.  However, the control group design (number-only
and string-only workloads) bounds the expected range.

\textbf{Endianness.}
Both x86-64 and arm64 targets are little-endian; big-endian behavior is
untested.  While the canonicalization algorithms do not depend on byte
order (they operate on IEEE~754 binary64 values via language-level
primitives), edge cases in binary representation parsing could
theoretically differ on big-endian platforms.

\textbf{CPU core affinity.}
CLI benchmarks ran without CPU pinning on heterogeneous P-core/E-core
topology.  Process migration may inflate variance for small workloads
near the noise floor.  API benchmarks (in-process) are less susceptible.

\textbf{Warmup validation.}
CLI benchmarks discard 3~warmup runs.  We did not formally validate
convergence; however, Go static binaries have minimal JIT-like warmup
effects, and permutation tests are robust to mild non-stationarity.

\textbf{Oracle SHA-256 circularity.}
SHA-256 checksums verify vector integrity but cannot verify oracle
correctness.  Independent verification against a second JS engine
(SpiderMonkey, JavaScriptCore) is noted as future work.

\textbf{sync.Pool mitigation.}
The \burgerdybvig implementation uses \texttt{sync.Pool} to reuse
big-integer buffers, reducing allocation cost.  Without pooling, the
performance gap would be wider.  This reflects realistic production usage
but may understate the theoretical algorithmic advantage of fixed-width
approaches.
