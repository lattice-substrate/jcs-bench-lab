\section{Methodology}
\label{sec:methodology}

\subsection{Conformance-Before-Performance}

Our methodology enforces a strict ordering: conformance verification
must pass before any performance data is collected.  This approach was
developed from practical experience: the standard pattern for \rfcjcs
number formatting (delegating to a language runtime's float-to-string
function) produces output that passes casual testing but fails \ecma
boundary-value verification (e.g., notation-switch thresholds at
$10^{21}$ and $10^{-7}$, even-digit tie-breaking under Note~2).  An
implementation that looks fast in a benchmark while producing wrong
output is unusable in a specification-constrained context
where the formatted bytes are cryptographically bound.  The citations
to Georges et al.~\cite{georges2007statistically} and Mytkowicz et
al.~\cite{mytkowicz2009producing} reinforce the approach; the
engineering experience originated it.

\textbf{Conformance gate:} 97 structural test cases per implementation
(776 total across 8~implementations) covering key ordering, Unicode
handling, numeric formatting, escape sequences, depth limits, and
invalid input rejection.  All implementations must produce byte-identical
output (verified via SHA-256) and identical error classification.

\textbf{Oracle vectors:} 286{,}362 IEEE~754 binary64 values with
expected \texttt{String(x)} output generated by the V8 JavaScript engine
(Node.js v23.3.0, V8 12.9.202.28) and independently validated against
SpiderMonkey (JavaScript-C115.10.0) and JavaScriptCore (WebKitGTK
2.50.4).  All three engines produce identical output for every vector,
confirming cross-engine agreement on ECMA-262 \texttt{Number::toString}
semantics.
(2{,}153 values appear in both vector files as overlapping boundary
coverage; the total counts file lines, with 284{,}209 unique values.)
Coverage includes all subnormals
with bit patterns $\leq 2^{12}$ (4{,}096 of approximately $4.5 \times 10^{15}$
positive subnormals; these provide dense coverage near zero where
notation-switch boundaries and leading-zero sequences create formatting
edge cases), subnormal/normal boundary ($\pm128$ around the transition),
all powers of two in the binary64 exponent range (binade boundaries
where the digit-generation algorithm must activate asymmetric interval
margins), \rfcjcs Appendix~B examples with $\pm2$ neighbors (catching
off-by-one errors in boundary comparisons against the specification's
own test values), and pseudorandom fill.

\subsection{Benchmark Design}

\textbf{CLI benchmarks:} end-to-end process execution measuring wall
time, CPU time (user + system), and peak RSS.  15~repetitions per
implementation/mode/workload combination, with 3~warmup runs discarded.
Task order is randomized with a fixed seed for reproducibility.

\textbf{Cross-language comparison pairs:} each of the four algorithms is
implemented in Go and Rust, yielding eight implementations.  These are
compared via 16~pairwise tests:
\begin{itemize}
\item 6~within-Go algorithm pairs (e.g., go:schubfach-vs-ryu,
      go:dragonbox-vs-bd),
\item 6~within-Rust algorithm pairs (e.g., rs:schubfach-vs-ryu,
      rs:dragonbox-vs-bd),
\item 4~cross-language same-algorithm pairs (e.g., schubfach:go-vs-rs,
      ryu:go-vs-rs).
\end{itemize}
The within-language pairs test the algorithmic effect while controlling
for language/runtime; the cross-language pairs test language/runtime
effects while controlling for algorithm.  Directional consistency across
within-language pairs in both Go and Rust provides evidence that
algorithmic differences are not solely attributable to language-specific
runtime characteristics, though this is not a
formal interaction test (see \S\ref{sec:discussion}).  The Rust
\burgerdybvig implementation uses the \texttt{num-bigint} crate (v0.4,
Karatsuba multiplication for large operands, heap-allocated with RAII
lifecycle), providing a production-grade multiprecision library
comparable to the Go \texttt{math/big}-based path.  Rust builds use release configuration
(\texttt{opt-level=3}, \texttt{lto=true}, \texttt{codegen-units=1}) to
align optimization policy with production benchmarking practice.

\textbf{API benchmarks:} Go \texttt{testing.B} microbenchmarks (Go
implementations) and a custom Rust benchmark harness
(\texttt{src/bin/bench.rs} in each Rust implementation) measuring
canonicalization and verify paths directly, 10~iterations per benchmark.
Go's \texttt{testing.B} auto-calibrates the number of inner iterations
per sample (typically hundreds to millions), whereas the Rust harness
times each function call individually ($N{=}1$) using
\texttt{std::time::Instant}; both report ns/op.  The Rust harness
includes 3~warmup iterations (discarded) before measurement to stabilize
caches and lazy-initialized data structures (e.g., the
\texttt{OnceLock}-based power-of-ten cache in the \burgerdybvig path).
The harness uses a counting global allocator
(\texttt{std::alloc::GlobalAlloc} with atomic counters) to report
per-operation allocation bytes and counts, and emits Go-compatible
\texttt{Benchmark... ns/op} lines so the same statistical pipeline can
be applied uniformly.  Low coefficients of variation ($<$5\%) are
achieved despite $N{=}1$ because the workload payloads are large enough
(0.1--14\,KB) that per-call latency is well above timer resolution.

\textbf{Measurement granularity note:} The Go and Rust API harnesses
differ in measurement granularity.  Go's \texttt{testing.B} averages
over many inner iterations per sample, producing low-variance samples.
The Rust harness reports single-call latency, producing higher-variance
samples.  Within-language comparisons are unaffected since both
implementations in a pair use the same harness.  Cross-language API
comparisons should be interpreted with this asymmetry in mind; we report
them for completeness but emphasize within-language results for
algorithmic conclusions.

\textbf{Differential fuzzing:} 2{,}000 randomly-generated JSON inputs
processed by all implementations, verifying byte-identical output.

\subsection{Workload Characterization}

For a JSON document $D$, we define $\nu(D)$ as the fraction of JSON
values that are numbers.  Every JSON value is counted once: primitives
(strings, numbers, booleans, null) and containers (arrays, objects);
container elements are counted separately.  For example,
\texttt{[1, "x", [2, 3]]} contains six values (two arrays, one string,
three numbers), so $\nu = 3/6 = 0.50$.

\subsection{Statistical Protocol}

All comparisons use:
\begin{itemize}
\item Permutation tests (5{,}000 resamples) for significance at
      $\alpha = 0.05$.
\item BCa (bias-corrected and accelerated) bootstrap 95\% confidence
      intervals (5{,}000 resamples) for speedup ratios.
\item Cohen's $d$ for effect size.\footnote{Computed as
      $(\bar{x}_A - \bar{x}_B) / s_p$ where $A$ and $B$ are the first
      and second implementation in each pair; negative values indicate
      implementation $A$ is faster.  In API benchmarks where coefficient
      of variation is typically ${<}$2\%, Cohen's $d$ values can exceed
      50 in absolute value; such magnitudes reflect low measurement
      variance rather than practically meaningful effect-size
      distinctions.  Readers should use the confidence intervals on
      speedup ratios as the primary measure of practical effect size;
      Cohen's $d$ is most informative in CLI benchmarks where higher
      variance produces values in the conventional interpretive range.}
\item Noise floor documentation: standard deviation of the faster
      implementation ($\sigma = \text{CV} \times \bar{x}_\text{faster}$),
      quantifying measurement noise in the same units as the metric.
\item Minimum detectable effect (MDE):
      $\frac{2.8 \sqrt{2}\,\times\,\text{CV}\,\times\,100}{\sqrt{N}}$,
      where $2.8 \approx z_{\alpha/2} + z_\beta$ for $\alpha = 0.05$ and
      80\% power (approximate MDE for a two-sample $z$-test with equal
      group sizes; uses only the faster implementation's CV and $n$ as an
      approximation).  This $z$-based constant slightly underestimates
      the true MDE for small $N$ (by ${\approx}3$--6\% compared to a
      $t$-based calculation at $N{=}10$--15).
\item Benjamini-Hochberg false discovery rate (FDR) control at $q = 0.05$
      across all 1{,}644~comparisons.  BH requires independence or positive
      regression dependency (PRDS); comparisons on the same workload across
      different pairs share one common implementation (e.g.,
      go:schubfach-vs-bd and go:ryu-vs-bd both include the BD
      implementation), inducing positive correlation and satisfying PRDS.  We report
      both raw and BH-adjusted $p$-values.
\item No outlier exclusion.  Permutation tests are nonparametric and
      robust to non-normality; BCa bootstrap accounts for skewness via the
      acceleration constant.
\item Permutation $p$-values have a resolution floor of
      $1/(R+1)$ where $R$ is the number of resamples; at $R = 5{,}000$
      the minimum reportable $p$-value is 0.0002.  Values reported at
      this floor should be interpreted as $p < 0.0002$.
\end{itemize}

Statistical resampling uses deterministic per-comparison seeds derived
via FNV-1a hashing of the comparison key, ensuring reproducible
$p$-values and confidence intervals across runs.

\subsection{Environment}

All performance benchmarks run on a single x86-64 machine with CPU
frequency governor set to \texttt{performance}, turbo boost state
documented, and load average recorded at benchmark start.  Full hardware
specification (CPU model, cache hierarchy, RAM, relevant CPU feature
flags) is captured automatically by the benchmark harness.

CPU core affinity pinning is supported (\texttt{-{}-pin-cpu}) but was not used
for the reported results.  The i7-12700K has heterogeneous P-core/E-core
topology; without pinning, the OS scheduler may migrate processes.  This
adds variance but does not bias the comparison since both implementations
share the same scheduling environment.

Cross-compilation determinism is verified by building arm64 binaries via
\texttt{GOARCH=arm64} and executing them under QEMU user-mode emulation
on the x86-64 host.  QEMU user-mode translates system calls but executes
floating-point operations on the host CPU; this tests Go compiler
determinism across target architectures, not native arm64 hardware
behavior.  Arm64 performance benchmarks are deferred to future work on
bare-metal hardware.
