\section{Conclusion}
\label{sec:conclusion}

We presented the first empirical comparison of multiprecision
(\burgerdybvig) and fixed-width (\schubfach) digit generation inside a
\rfcjcs/\ecma-constrained JSON canonicalization pipeline, with
cross-language validation in Go and Rust to separate algorithmic effects
from language/runtime characteristics.  While prior work has compared
float-to-string algorithms in isolation (e.g., Adams's comparison of
Ry\=u against Grisu and Errol), our study is the first to evaluate these
algorithms within a specification-constrained canonicalization pipeline
using conformance-gated methodology.

Key findings:
\begin{itemize}
\item \schubfach achieves 1.3--1.7$\times$ speedup on number-dense
      workloads ($\nu(D) \geq 40\%$) in Go API benchmarks due to zero
      heap allocation; the advantage replicates at 1.6--4.2$\times$ in
      Rust, confirming it is algorithmic.  On a small number of API-level
      non-number workloads, \burgerdybvig is marginally faster.
\item In end-to-end Go CLI measurement, 12 of 38 comparisons reach
      BH-corrected significance with speedups of 1.2--1.5$\times$.
\item The advantage narrows to $<$5\% on string-dominant and
      deeply-nested inputs.
\item Go implementations produce byte-identical output under
      cross-compilation for x86-64 and arm64 (arm64 binaries verified
      via QEMU user-mode emulation on x86-64 host).  Rust
      implementations were not cross-compiled in this study.
\item Preliminary profiling suggests digit generation accounts for
      approximately 15--40\% of total canonicalization cost depending
      on number density.
\end{itemize}

The benchmark infrastructure, 286{,}362 oracle vectors, and all raw data
are released as a reproducibility artifact.  Our findings align with the
broader ecosystem direction: Go~1.26 adopted the Dragonbox fixed-width
algorithm for \texttt{strconv}, corroborating our evidence that
fixed-width approaches offer meaningful performance advantages in
float-to-string formatting.

\subsection*{Future Work}

Bare-metal arm64 performance benchmarks, additional language
implementations (C and JVM) to extend the comparison matrix to unmanaged
and managed-GC runtimes, multi-engine oracle validation (SpiderMonkey,
JavaScriptCore), Rust cross-compilation verification, and workload
characterization from production JSON corpora.
