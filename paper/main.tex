\title{Fixed-Width vs.\ Multiprecision Digit Generation\\for JSON Canonicalization: A Four-Algorithm Empirical Study}

\author{Mark Lenhardt}
\affil{Lattice Substrate}
\date{}

\maketitle

\begin{abstract}
Benchmarks that measure only speed are insufficient when the code
under test must satisfy a formatting specification: a fast but
non-conformant implementation is worse than useless.  We introduce
\emph{conformance-gated benchmarking}, a methodology that treats
specification correctness as a binary prerequisite for performance
measurement.  The methodology gates every candidate on an oracle
vector suite, documents noise floors and minimum detectable effect
sizes for every comparison, and controls false-discovery rate across
the full comparison matrix.  It is general---applicable wherever
correctness is a hard prerequisite---and was developed from the
practical discovery that existing JSON Canonicalization Scheme
(RFC~8785) implementations delegate number formatting to language
runtimes that silently violate the required ECMA-262
\texttt{Number::toString} policy.

We apply the methodology to a case study comparing four
digit-generation algorithms---three fixed-width (\schubfach, \ryualg,
\dragonbox) and one multiprecision (\burgerdybvig)---inside an
RFC~8785 canonicalization pipeline, with cross-language validation in
Go and Rust (8~implementations, 1{,}644 statistical comparisons).
Conformance is verified against 286{,}362 oracle vectors generated
from V8 and validated against SpiderMonkey and JavaScriptCore.
All three fixed-width algorithms perform within 1--8\% of each other
across both languages, while each is 1.4--1.8$\times$ faster than
\burgerdybvig on number-dense workloads ($\nu(D) \geq 40\%$) in Go
API benchmarks; these pipeline-level ratios are consistent with
larger isolation-level speedups attenuated by Amdahl's Law, since
digit generation accounts for approximately 15--40\% of total
pipeline cost; the advantage replicates at 2.3--3.0$\times$ in Rust
(excluding two pure-numeric workloads producing 8--9$\times$ ratios
discussed in \S\ref{sec:discussion}).
On string-dominant and deeply-nested inputs the difference narrows to
$<$5\%.
All eight implementations produce byte-identical output; Go binaries are
additionally verified under cross-compilation for x86-64 and arm64.
We release the benchmark infrastructure, oracle vectors, and all raw
data as a reproducibility artifact.
\end{abstract}

\section*{Data Availability Statement}

All data, software, and oracle vectors are freely available online.
The benchmark infrastructure, all eight implementations, workload manifests,
and oracle generation scripts are available at the accompanying
repository.\footnote{\url{https://github.com/lattice-substrate/jcs-bench-lab}.
Zenodo DOI to be minted before camera-ready submission.}  Complete benchmark tables (full API benchmarks,
full CLI benchmarks, and all 1{,}644 statistical comparisons across
16~pairs) are provided in the supplementary materials.  Oracle vector
checksums (SHA-256) are documented in \texttt{PROVENANCE.json} files
accompanying each implementation.
